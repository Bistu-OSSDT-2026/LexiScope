# 贡献指南（Contributing Guide）

感谢你对 **LexiScope** 项目的兴趣！本文档旨在规范本项目的协作流程，确保每一位成员都能高效、有序地参与开发。

本指南遵循课程《项目开发与小组协作建议流程》中的约定，采用 **Issue 驱动 + PR 审查** 的经典开源协作模式。

---

## 1. 核心协作流程

所有功能开发、Bug修复和文档完善，必须严格按照以下流程进行：

1. **创建或认领 Issue**：在 GitHub Issues 中查找任务，或提出新任务并等待确认。
2. **指派给自己**：确认认领后，将 Issue 的负责人（Assignee）设为自己。
3. **创建开发分支**：基于 `main` 分支，新建一个命名规范的分支（见第 2 节）。
4. **本地开发与自测**：在分支上完成代码，并确保本地运行无误。
5. **提交 Pull Request (PR)**：推送分支到远程仓库，并创建 PR 请求合并回 `main`。
6. **成员审查 (Review)**：至少由 **1 名其他成员** 进行代码审查。
7. **修改与合并**：根据审查意见修改，通过后由负责人合并分支，并关闭对应的 Issue。

---

## 2. 分支命名规范

为了便于追溯，分支命名请遵循以下格式：

- **新功能开发**：`feature/功能简述`（例如 `feature/word-cloud`）
- **Bug 修复**：`fix/问题简述`（例如 `fix-encoding-error`）
- **文档/配置更新**：`docs/修改内容`（例如 `docs/update-readme`）

---

## 3. 代码规范（极其重要）

为了保证最终项目能够顺利合并为一个完整的 `app.py` 文件并正常运行，**所有贡献者必须严格遵守以下统一的代码契约**：

### 3.1 函数签名（输入格式）

所有核心处理函数**只能有 1 个参数**，参数名必须叫 `text`，类型为字符串。

```python
# ✅ 正确
def get_word_freq(text): ...

# ❌ 错误（参数多了 / 参数名不对）
def get_word_freq(text, stopwords): ...
def extract_keywords(file_path): ...
```

### 3.2 输出格式（返回值）

每个模块必须返回指定结构的字典，不能返回 `None` 或列表。

| 模块 | 函数名 | 返回格式示例 |
| :--- | :--- | :--- |
| 词频统计 | `get_word_freq` | `{"人工智能": 15, "算法": 12}` （Top20） |
| 关键词提取 | `extract_keywords` | `{"算法": 0.856, "模型": 0.721}` （Top10，保留3位小数） |
| 情感分析 | `get_sentiment` | `{"score": 0.35, "label": "积极", "pos": 12, "neg": 5}` |

### 3.3 依赖隔离

- **仅 B 同学（词频）** 可以使用 `jieba` 库。
- **仅 A 同学（后端）** 可以使用 `Flask` 库。
- **C 同学（TF-IDF）和 D 同学（情感）** 只允许使用 Python 内置库（`re`、`math`、`collections` 等），防止环境不一致导致运行失败。

### 3.4 代码分区

每个人的代码必须用以下注释标记包裹，方便最终合并：

```python
# ============================================
# === 队员X：模块名称（函数：function_name）===
# ============================================
def function_name(text):
    # 你的代码...
# ============================================
# === 队员X 模块结束 ===
# ============================================
```

## 4. Pull Request 审查标准

当成员提交 PR 后，审查者（Reviewer）需要重点检查以下内容：

1. **功能完整性**：是否完成了 Issue 中描述的所有要求？
2. **可运行性**：代码能否在本地正常启动并运行？
3. **契约符合度**：
   - 函数名是否完全正确（`get_word_freq` / `extract_keywords` / `get_sentiment`）？
   - 返回值格式是否与上述规范 3.2 中的字典结构完全一致？
   - 是否处理了空文本等极端情况（返回空字典或默认情感值）？
4. **代码整洁度**：是否有调试用的 `print` 语句或死代码（注释掉的旧逻辑）？

---

## 5. Issue 创建规范

在创建新 Issue 时，建议包含以下信息，以便成员快速理解任务：

- **标题**：简洁描述要做什么（如“增加对空文件的错误提示”）。
- **内容**：
  - **背景/动机**：为什么需要这个改动？
  - **验收标准**：完成之后应该达到什么效果（例如“上传空文件时，前端显示红色提示框”）？
  - **指派人**：建议直接 @ 相关负责人。

## 7. 版本发布流程

当项目开发进入收尾阶段，应停止新增功能，集中进行测试与文档完善。完成后：

1. 确保 `main` 分支所有自动化检查（CI）通过。
2. 更新 `README.md` 中的使用说明。
3. 在 GitHub 上创建 **Tag** 并发布 **Release**（版本号如 `v0.1.0`）。
4. Release Notes 中需包含：本版本功能清单、使用方法、已知问题（如果有）。
